## はじめに

Claude Architectになり、プロダクショングレードのアプリケーションを構築するには、以下の4つを深く理解する必要がある：

1. **Claude Code**
2. **Claude Agent SDK**
3. **Claude API**
4. **Model Context Protocol（MCP）**

これはClaude Certified Architect試験の内容に基づいている。試験を受けるにはClaudeパートナーである必要があるが、**資格の有無は関係ない**。この試験で問われる知識を身につけること自体が、プロダクショングレードのアプリケーションを構築する力になる。

証明書がなくても、プロダクショングレードのアプリケーションは構築できる。必要なのは**知識**だけだ。

本ガイドは試験ガイド全体から実際に重要な部分を抽出・解説したものである。

## 試験が測る6つの実践スキル

試験で扱われるのは理論的な演習ではなく、本番AIシステムの構築に使われる実際のパターンである：

- **カスタマーサポート解決エージェント**：Agent SDK + MCP + エスカレーションワークフロー
- **コード生成システム**：CLAUDE.md設定 + プランモード + スラッシュコマンド
- **マルチエージェントリサーチシステム**：コーディネーター＝サブエージェントのオーケストレーション
- **開発者生産性ツール**：組み込みツール + MCPサーバー
- **CI/CDパイプライン内でのClaude Code**：非対話型パイプライン + 構造化出力
- **構造化データ抽出パイプライン**：JSONスキーマ + `tool_use` + バリデーションループ

## ドメイン1：エージェンティックアーキテクチャとオーケストレーション（27%）

試験全体で**最も配点が高い**ドメイン。

試験では以下の3つのアンチパターンを即座に排除する必要がある：
- 自然言語のパースでループ終了を判定する
- 任意のイテレーション上限を主要な停止メカニズムとする
- アシスタントのテキスト応答を完了インジケータとして使う

**最大の誤解**：サブエージェントがコーディネーターとメモリを共有していると思い込むこと。サブエージェントは分離されたコンテキストで動作する。すべての情報はプロンプト内で明示的に渡す必要がある。

**最も重要なルール**：金融やセキュリティに関わる高リスクな場面では、プロンプトの指示だけでは不十分。フックや前提条件ゲートによるプログラム的な強制が必要。

### タスクステートメント1.1：エージェンティックループ

エージェンティックループの完全なライフサイクル：

1. Messages APIでClaudeにリクエストを送信
2. レスポンスの`stop_reason`フィールドを検査
3. `stop_reason`が`"tool_use"`の場合：リクエストされたツールを実行 → ツール結果を会話履歴に追加 → 更新された会話をClaudeに再送信
4. `stop_reason`が`"end_turn"`の場合：エージェント完了、最終レスポンスを提示
5. ツール結果は会話履歴に追加する必要がある（モデルが次のイテレーションで新情報を推論できるように）

**3つのアンチパターン**：
- **自然言語パースによるループ終了判定**：「完了しました」等のアシスタントの発言をチェック → 自然言語は曖昧で信頼性がない。`stop_reason`フィールドがまさにこの目的で存在する
- **任意のイテレーション上限**：「10回ループしたら停止」→ 有用な処理を途中で切るか、不要なイテレーションを実行する。モデルは`stop_reason`で完了を通知する
- **アシスタントテキストの存在チェック**：「レスポンスにテキストがあれば完了」→ モデルは`tool_use`ブロックと同時にテキストを返すことがある

**モデル駆動の意思決定 vs 固定デシジョンツリー**の違い：Claudeがコンテキストに基づいてどのツールを呼ぶか推論する（モデル駆動）vs 事前定義された手順（デシジョンツリー）。試験はモデル駆動のアプローチを柔軟性の点で好むが、重要なビジネスロジックにはプログラム的な強制を求める（1.4で詳述）。

### タスクステートメント1.2：マルチエージェントオーケストレーション

**ハブ＆スポークアーキテクチャ**：
- **コーディネーターエージェント**が中心に位置
- **サブエージェント**（スポーク）が特定タスクを実行
- **すべての通信はコーディネーターを経由**。サブエージェント同士が直接通信することはない
- コーディネーターの責任：タスク分解、サブエージェント選択、コンテキスト受け渡し、結果集約、エラーハンドリング、情報ルーティング

**重要な分離原則**：
- サブエージェントはコーディネーターの会話履歴を自動的に引き継が**ない**
- サブエージェントは呼び出し間でメモリを共有**しない**
- サブエージェントが必要とするすべての情報は、プロンプト内に明示的に含める必要がある
- **マルチエージェントシステムで最も誤解されているコンセプト**

**コーディネーターの責務**：
- クエリ要件を分析し、呼び出すサブエージェントを動的に選択（常にパイプライン全体を通すわけではない）
- サブエージェント間でリサーチ範囲を分割し、重複を最小化（異なるサブトピックやソースタイプを割り当て）
- 反復的な改善ループ：合成出力のギャップを評価 → 的を絞ったクエリで再委任 → 十分なカバレッジまで再呼び出し

**狭い分解の失敗**：
- 「AIがクリエイティブ産業に与える影響」を視覚芸術のサブトピックだけに分解し、音楽・執筆・映画を完全に見落とす例
- 根本原因はコーディネーターの分解であり、下流のエージェントではない
- 試験は失敗を原因まで遡って追跡することを求める

### タスクステートメント1.3：サブエージェントの呼び出しとコンテキスト受け渡し

**Taskツール**：
- コーディネーターからサブエージェントを生成するメカニズム
- コーディネーターの`allowedTools`に"Task"が含まれている必要がある
- 各サブエージェントはAgentDefinition（description、system prompt、ツール制限）を持つ

**コンテキスト受け渡し**：
- 先行エージェントの完全な調査結果をサブエージェントのプロンプトに直接含める
- コンテンツとメタデータ（ソースURL、ドキュメント名、ページ番号）を分離する構造化データ形式を使用し、エージェント間で帰属情報を保持
- コーディネータープロンプトはリサーチ目標と品質基準を指定する（手順的な指示ではなく）

**並列生成**：
- コーディネーターの単一レスポンスで複数のTaskツール呼び出しを発行し、サブエージェントを並列生成
- 逐次呼び出しより高速
- 試験はレイテンシの意識をテストする

**`fork_session`**：
- 共有された分析ベースラインから独立したブランチを作成
- 異なるアプローチの探索に使用（例：同じコードベース分析から2つのテスト戦略を比較）

### タスクステートメント1.4：ワークフロー強制とハンドオフ

**強制のスペクトラム**：
- **プロンプトベースのガイダンス**：「必ず顧客を先に確認してください」等の指示 → ほとんどの場合機能するが、ゼロではない失敗率
- **プログラム的な強制**：フックや前提条件ゲートで物理的にダウンストリームツールをブロック → 100%機能

**試験の判断ルール**：
- 結果が金融・セキュリティ・コンプライアンスに関わる場合 → **プログラム的な強制**を使用
- 結果が低リスク（フォーマット設定、スタイルガイドライン）の場合 → プロンプトベースのガイダンスで可
- 試験は高リスクシナリオにプロンプトベースの解決策を選択肢として提示する → **拒否すべき**

**構造化ハンドオフプロトコル**：
- 人間のエージェントにエスカレーションする際のコンパイル項目：顧客ID、会話サマリー、根本原因分析、返金額（該当する場合）、推奨アクション
- 人間のエージェントは会話トランスクリプトにアクセスできない
- ハンドオフサマリーは自己完結的でなければならない

### タスクステートメント1.5：Agent SDKフック

**PostToolUseフック**：
- ツール実行後、モデルが処理する前にツール結果をインターセプト
- ユースケース：異なるMCPツールからの異種データフォーマットを正規化（UnixタイムスタンプをISO 8601に、数値ステータスコードを人間が読める文字列に）

**ツール呼び出しインターセプションフック**：
- 実行前に発信ツール呼び出しをインターセプト
- ユースケース：$500以上の返金をブロックして人間のエスカレーションワークフローにリダイレクト
- ユースケース：コンプライアンスルールの強制（マネージャー承認の要求等）

**判断フレームワーク**：
- **フック = 決定論的な保証**。100%遵守が必要なビジネスルールに使用
- **プロンプト = 確率的なガイダンス**。好みやソフトルールに使用
- 単一の失敗でビジネスが損失や法的リスクに直面する場合 → フックを使用

### タスクステートメント1.6：タスク分解戦略

**固定順次パイプライン（プロンプトチェイニング）**：
- 作業を事前定義された順次ステップに分割
- 例：各ファイルを個別に分析 → クロスファイル統合パスを実行
- 最適：コードレビュー、ドキュメント処理など予測可能で構造化されたタスク
- 利点：一貫性と信頼性 / 制限：予期しない発見に適応できない

**動的適応分解**：
- 各ステップで発見された内容に基づいてサブタスクを生成
- 例：「レガシーコードベースにテストを追加」→ 構造のマッピング → 高インパクト領域の特定 → 依存関係に応じた優先付き計画
- 最適：オープンエンドな調査タスク
- 利点：問題に適応 / 制限：予測性が低い

**注意力の希釈問題**：
- 単一パスで多数のファイルを処理すると、深さにムラが出る
- 修正：ファイルごとのローカル分析パス＋別途クロスファイル統合パス

### タスクステートメント1.7：セッション状態と再開

**セッション管理オプション**：
- `--resume <session-name>`：特定の名前付きセッションを続行
- `fork_session`：共有ベースラインから独立ブランチを作成
- **サマリー注入で新規開始**：新しいセッションを開始し、先行調査結果の構造化サマリーを初期コンテキストに注入

**使い分け**：
- **再開**：先行コンテキストがまだ有効で、ファイルが大きく変わっていない場合
- **フォーク**：共有分析ポイントから異なるアプローチを探索する場合
- **新規開始**：ツール結果が古い、ファイルが変更された、長いセッションでコンテキストが劣化した場合

**古いコンテキスト問題**：
- コード変更後にセッションを再開すると、モデルが古いツール結果から推論して矛盾したアドバイスをする
- 修正：具体的なファイル変更をエージェントに通知して的を絞った再分析を行う
- サマリー注入で新規開始する方が、古いツール結果で再開するより信頼性が高い

## ドメイン2：ツール設計とMCPインテグレーション（18%）

ツールの説明（description）は非常に見落とされがちだが、試験はこれを重点的にテストする。

ツールの説明はClaudeがツール選択に使う**主要なメカニズム**である。説明が曖昧だったり重複していると、選択が信頼性を失う。

サンプル問題：`get_customer`と`lookup_order`がほぼ同一の説明で常に誤ルーティング。正しい修正はfew-shotでもルーティング分類器でもツール統合でもない。**より良い説明を書くこと**。

`tool_choice`オプションを完全に把握：`"auto"`（テキストを返す可能性あり）、`"any"`（必ずツールを呼ぶが選択はモデル）、強制選択（特定ツールを必ず呼ぶ）。

エージェントに18個のツールを与えると選択の信頼性が低下する。各サブエージェントに**4-5個のツール**に絞る。

- MCP Integration for Claude Code：サーバースコーピング、環境変数展開、プロジェクト vs ユーザー設定
- MCP仕様とコミュニティサーバー：プロトコルの理解、コミュニティサーバー vs カスタムビルドの判断

### タスクステートメント2.1：ツールインターフェース設計

ツールの説明はLLMがツール選択に使う**最重要メカニズム**。補助的なものではない。**それ自体がメカニズム**。

**良いツール説明に含めるもの**：
- ツールが何をするか（主な目的）
- 期待される入力（形式、型、制約）
- うまく処理できるクエリの例
- エッジケースと制限
- 明示的な境界：このツール vs 類似ツールの使い分け

**誤ルーティング問題**：
- 重複する説明を持つ2つのツール → 選択に混乱
- 修正：説明を充実させる。few-shot（間違った根本原因へのトークンオーバーヘッド）でも、ルーティング分類器（初手として過剰設計）でも、ツール統合（労力過多）でもない

**ツール分割**：
- 汎用ツールを目的別ツールに分割し、入出力契約を定義
- 例：`analyze_document` → `extract_data_points`、`summarize_content`、`verify_claim_against_source`に分割

### タスクステートメント2.2：構造化エラーレスポンス

MCPの`isError`フラグパターンでエージェントに失敗を通知。

**4つのエラーカテゴリ**：
- **一時的（Transient）**：タイムアウト、サービス停止 → リトライ可能
- **バリデーション（Validation）**：無効な入力 → 入力を修正してリトライ
- **ビジネス（Business）**：ポリシー違反（返金が上限超過）→ リトライ不可、代替ワークフローが必要
- **権限（Permission）**：アクセス拒否 → エスカレーションまたは別の認証情報が必要

**重要な区別**：
- **アクセス失敗**：ツールがデータソースに到達できなかった → リトライを検討
- **有効な空結果**：ツールがソースにクエリして一致なし → リトライ不要。答えは「結果なし」
- この2つを混同するとリカバリーロジックが壊れる

### タスクステートメント2.3：ツール配分と`tool_choice`

**ツール過負荷問題**：
- 18個のツールを与えると選択の信頼性が低下
- 最適：エージェントの役割に応じた**4-5個のツール**
- 合成エージェントにWeb検索ツールを持たせるべきでない。Web検索エージェントにドキュメント分析ツールを持たせるべきでない

**`tool_choice`設定**：
- `"auto"`：ツールを呼ぶかテキストを返すかモデルが決定。デフォルト
- `"any"`：必ずツールを呼ぶが、どれを呼ぶかはモデルが選択。複数スキーマからの構造化出力の保証に使用
- `{"type": "tool", "name": "extract_metadata"}`：特定ツールを必ず呼ぶ。強制的な初期ステップに使用

**スコープ付きクロスロールツール**：
- 高頻度の単純操作には、制約付きツールを直接エージェントに渡す
- 例：合成エージェントに簡単な検証用の`verify_fact`ツールを与え、複雑な検証はコーディネーター経由
- 85%が単純なケースの場合、コーディネーターへの往復レイテンシを回避

### タスクステートメント2.4：MCPサーバーインテグレーション

**スコーピング階層**：
- **プロジェクトレベル**：`.mcp.json`（リポジトリ内）。バージョン管理あり。チーム共有
- **ユーザーレベル**：`~/.claude.json`。個人用。バージョン管理なし。共有されない
- 設定された全サーバーのすべてのツールが接続時に検出され、同時に利用可能

**環境変数展開**：
- `.mcp.json`は`${GITHUB_TOKEN}`構文をサポート
- 認証情報をバージョン管理外に保持
- 各開発者がローカルで自分のトークンを設定

**MCPリソース**：
- コンテンツカタログ（issueサマリー、ドキュメント階層、データベーススキーマ）をMCPリソースとして公開
- 探索的なツール呼び出しなしにエージェントが利用可能なデータを把握できる

**構築 vs 利用の判断**：
- 標準的なインテグレーション（Jira、GitHub、Slack）には既存のコミュニティMCPサーバーを使用
- チーム固有のワークフローでコミュニティサーバーが対応できない場合のみカスタムサーバーを構築

### タスクステートメント2.5：組み込みツール

**Grep vs Glob**の違い：
- **Grep**：ファイルの**内容**をパターン検索。用途：関数の呼び出し元検索、エラーメッセージの特定、import文の検索
- **Glob**：ファイルの**パス**を名前パターンでマッチ。用途：拡張子によるファイル検索（`**/*.test.tsx`）、設定ファイルの特定

**Read/Write/Edit**：
- **Edit**：一意なテキストマッチングによる部分修正。高速・正確
- Editが失敗（一意でないテキスト）した場合：**Read**（全ファイル読み込み）+ **Write**（完全な修正ファイルを書き込み）にフォールバック

**段階的なコードベース理解**：
- まずGrepでエントリーポイントを見つける（関数定義、import文）
- Readでインポートを辿ってフローを追跡
- **全ファイルを先に読まない**。コンテキスト予算の浪費になる

## ドメイン3：Claude Codeの設定とワークフロー（20%）

Claude Codeを**使う人**と、チーム用に**設定した人**を分けるドメイン。

### タスクステートメント3.1：CLAUDE.md階層

**3つのレベル**：
1. **ユーザーレベル**（`~/.claude/CLAUDE.md`）：自分だけに適用。バージョン管理されない。git経由で共有されない。リポジトリをクローンした新メンバーにはこの指示が反映されない
2. **プロジェクトレベル**（`.claude/CLAUDE.md` またはルートCLAUDE.md）：全員に適用。バージョン管理あり。チーム全体の標準をここに配置
3. **ディレクトリレベル**（サブディレクトリのCLAUDE.md）：そのディレクトリで作業しているときに適用

**試験の定番トラップ**：新しいチームメンバーが指示を受け取っていない → 原因：指示がユーザーレベルに保存されている（バージョン管理されず共有されない）

**モジュラー構成**：
- `@import`構文でCLAUDE.mdから外部ファイルを参照
- `.claude/rules/`ディレクトリにトピック別ルールファイル（testing.md、api-conventions.md、deployment.md）→ 巨大な単一ファイルの代替

### タスクステートメント3.2：カスタムスラッシュコマンドとスキル

**ディレクトリ構造**：
- `.claude/commands/` = プロジェクトスコープ、バージョン管理で共有
- `~/.claude/commands/` = 個人用、共有されない
- `.claude/skills/` + SKILL.mdファイル = オンデマンド呼び出し、設定付き

**スキルfrontmatterオプション**：
- `context: fork`：分離されたサブエージェントコンテキストで実行。冗長な出力を封じ込め、メインの会話をクリーンに保つ
- `allowed-tools`：スキルが使用できるツールを制限
- `argument-hint`：引数なしで呼び出された場合に必要なパラメータを要求

**重要な区別**：
- **スキル** = オンデマンド、タスク特化のワークフロー（必要時に呼び出し）
- **CLAUDE.md** = 常時ロード、普遍的な標準（自動適用）
- タスク固有の手順をCLAUDE.mdに置かない。普遍的な標準をスキルに置かない

### タスクステートメント3.3：パス固有ルール

`.claude/rules/`ファイル + YAMLフロントマター：

```yaml
---
paths: ["terraform/**/*"]
---
```

グロブパターンにマッチするファイルを編集している時のみルールがロードされる。

**ディレクトリレベルCLAUDE.mdに対する利点**：
- グロブパターンは**コードベース全体**にマッチ
- `**/*.test.tsx`はディレクトリに関係なくすべてのテストファイルをキャッチ
- ディレクトリレベルCLAUDE.mdはそのディレクトリ内のファイルにのみ適用

**トークン効率**：
- パススコープルールはマッチするファイルの編集時のみロード
- 常時ロードの指示と比べ、無関係なコンテキストとトークン使用量を削減

### タスクステートメント3.4：プランモード vs 直接実行

**プランモード**を使う場面：
- 大規模な変更を伴う複雑なタスク
- 複数の有効なアプローチがある（コミット前に評価が必要）
- アーキテクチャ上の判断が必要
- マルチファイル修正（45以上のファイルに影響するライブラリ移行）

**直接実行**を使う場面：
- 明確で限定的なスコープのよく理解された変更
- 明確なスタックトレースのある単一ファイルのバグ修正
- 日付バリデーションの条件追加

**Exploreサブエージェント**：
- 冗長な調査出力をメインの会話から分離
- サマリーを返してメインの会話コンテキストを保持
- コンテキストウィンドウの枯渇を防ぐためにマルチフェーズタスクで使用

### タスクステートメント3.5：反復的な改善

**テクニック階層**：
- **具体的な入出力例**（2-3例のbefore/after）：散文的な説明を常に上回る
- **テスト駆動反復**：テストを先に書き、失敗を共有して改善をガイド
- **インタビューパターン**：実装前にClaudeに質問させる（不慣れなドメインで見落とす考慮事項を表面化）

**バッチ vs シーケンシャルフィードバック**：
- **単一メッセージ**：修正が相互に影響する場合
- **逐次反復**：問題が独立している場合

### タスクステートメント3.6：CI/CDインテグレーション

**`-p`フラグ**：
- Claude Codeを非対話モード（プリントモード）で実行
- なしだとCIジョブが対話入力待ちでハングする

**構造化CI出力**：
- `--output-format json` + `--json-schema`：機械解析可能な構造化結果を生成
- 自動化システムがインラインPRコメントとして投稿可能

**セッションコンテキスト分離**：
- コードを生成した同じClaudeセッションは、自身の変更のレビューに**効果が低い**
- 推論コンテキストを保持しているため、自分の判断を疑う可能性が低い
- コードレビューには**独立したレビューインスタンス**を使用

**インクリメンタルレビューコンテキスト**：
- 新しいコミット後にレビューを再実行する場合、先行レビューの結果をコンテキストに含める
- 新規または未対応の問題**のみ**を報告するよう指示
- 重複コメントを防ぎ、開発者の信頼を維持

### 学習リソース

- Claude Code公式ドキュメント：CLAUDE.md階層、rulesディレクトリ、スラッシュコマンド、skills frontmatter
- Claude Code CLIチートシート：コマンド、スキル、フック、CI/CDフラグの実践リファレンス
- Creating the Perfect CLAUDE.md：チーム設定パターンとMCPインテグレーション

## ドメイン4：プロンプトエンジニアリングと構造化出力（20%）

このドメイン全体を貫くキーワード：**明示的であれ**。

「保守的に」では精度は上がらない。「高信頼度の結果のみ報告」では偽陽性は減らない。効果的なのは：**どの問題を報告し、どの問題をスキップするか**を正確に定義し、各重要度レベルの具体的なコード例を添えること。

**few-shotの例は、テストされる最もレバレッジの高いテクニック**。2-4個の的を絞った例で、曖昧なケースの処理と、なぜある行動を他の選択肢より選んだかの推論を示す。

`tool_use` + JSONスキーマは構文エラーを排除する。ただし**意味的なエラーは排除しない**。スキーマ設計：ソースデータが欠落している可能性がある場合のnullableフィールド（捏造を防止）、"unclear"列挙値、"other" + 詳細文字列。

**Message Batches API**：50%コスト削減、最大24時間の処理、レイテンシSLAなし、マルチターンのツール呼び出しなし。一晩のレポートにはバッチ。マージ前のブロッキングチェックには同期API。

### 学習リソース

- Anthropicプロンプトエンジニアリングドキュメント：few-shotパターン、明示的基準、構造化出力
- Anthropic APIツール使用ドキュメント：`tool_use`、`tool_choice`設定、JSONスキーマ強制
- 試験ガイドのサンプル問題（Q10、Q11、Q12）：このドメインの最良の学習教材

### タスクステートメント4.1：明示的な基準

**核心原則**：具体的なカテゴリ基準は、曖昧な信頼度ベースの指示を圧倒する。

- NG：「保守的にしてください」「高信頼度の結果のみ報告してください」
- OK：「コメントで主張される動作と実際のコード動作が矛盾する場合のみフラグ。バグとセキュリティ脆弱性を報告。軽微なスタイル設定とローカルパターンはスキップ」

**偽陽性の信頼問題**：
- 一つのカテゴリで偽陽性率が高いと、**すべてのカテゴリ**の信頼が崩壊する
- 修正：高偽陽性カテゴリを一時的に無効化し、プロンプトを改善しながら信頼を回復

**重要度キャリブレーション**：
- 各レベルの具体的な**コード例**で明示的な重要度基準を定義
- 散文的な重要度説明ではなく、「クリティカル」vs「マイナー」が実際のコードでどう見えるかを示す

### タスクステートメント4.2：few-shotプロンプティング

few-shotの例は**一貫性のための最も効果的なテクニック**。追加の指示でも、信頼度閾値でもない。

**デプロイすべき時**：
- 詳細な指示だけでは一貫性のないフォーマットになる
- モデルが曖昧なケースで一貫性のない判断をする
- 抽出タスクでドキュメントに存在する情報に対してempty/nullフィールドが生成される

**構成方法**：
- 曖昧なシナリオに対する2-4個の的を絞った例
- 各例はなぜある行動をもっともらしい代替案より選んだかの**推論**を示す
- パターンマッチングではなく、新しいパターンへの一般化を教える

### タスクステートメント4.3：`tool_use`による構造化出力

**信頼性の階層**：
- `tool_use` + JSONスキーマ = 構文エラーを完全に排除
- プロンプトベースJSON = 不正なJSONの可能性あり

**`tool_use`が防止しないもの**：
- **意味的エラー**：明細の合計が記載総額と一致しない
- **フィールド配置エラー**：値が間違ったフィールドに入る
- **捏造**：ソースに情報がない必須フィールドにモデルが値を発明

**スキーマ設計**：
- ソースに情報がない可能性がある場合はoptional/nullableフィールド → **捏造を防止**
- 曖昧なケース用の"unclear"列挙値
- 拡張可能なカテゴリ用の"other" + 自由記述の詳細文字列

### タスクステートメント4.4：バリデーション-リトライループ

**エラーフィードバック付きリトライ**：
- 送り返す情報：元のドキュメント + 失敗した抽出 + 具体的なバリデーションエラー
- モデルがエラーを使って自己修正

**リトライの有効性境界**：
- **有効**：フォーマットの不一致、構造的な出力エラー、値の誤配置
- **無効**：ソースドキュメントに本当に情報がない場合
- 試験は両方のシナリオを提示する。どちらが修正可能かを見極める必要がある

**`detected_pattern`フィールド**：
- 構造化結果にどのコード構造がトリガーしたかを追跡するフィールドを追加
- 開発者が結果を却下した際のパターン分析を可能にし、プロンプトを体系的に改善

### タスクステートメント4.5：バッチ処理

**Message Batches APIの制約**：
- 50%のコスト削減
- 最大24時間の処理ウィンドウ
- レイテンシSLAなし
- 単一リクエスト内のマルチターンツール呼び出し非対応
- `custom_id`でリクエスト/レスポンスを紐付け

**マッチングルール**：
- **同期API**：ブロッキングワークフロー（マージ前チェック、開発者が待つもの）
- **Batch API**：レイテンシ許容ワークフロー（一晩のレポート、週次監査、夜間テスト生成）

**バッチ失敗ハンドリング**：
- `custom_id`で失敗したドキュメントを特定
- 失敗分のみ修正して再送信（例：大きすぎるドキュメントのチャンキング）
- バッチ処理前にサンプルセットでプロンプトを改善し、初回成功率を最大化

### タスクステートメント4.6：マルチインスタンスレビュー

**セルフレビューの限界**：
- 同じセッションで自分の出力をレビューするモデルは推論コンテキストを保持
- 自分の判断を疑う可能性が低い
- 先行コンテキストなしの独立インスタンスの方が微妙な問題を発見する

**マルチパスアーキテクチャ**：
- **ファイルごとのローカル分析パス**：ファイルごとに一貫した深さ
- **別途クロスファイル統合パス**：ファイル間のデータフローの問題をキャッチ
- 注意力の希釈と矛盾する結果を防止

## ドメイン5：コンテキスト管理と信頼性（15%）

配点は最小。しかしここでの間違いは**すべてに波及**する。

### タスクステートメント5.1：コンテキスト保持

**プログレッシブ要約のトラップ**：
- 会話履歴を圧縮すると、数値・日付・パーセンテージ・顧客の期待が曖昧なサマリーに変わる
- 「顧客は注文#8891の$247.83の返金を希望（3月3日注文）」→ 「顧客は最近の注文の返金を希望」
- 修正：トランザクション事実を永続的な「ケースファクト」ブロックに抽出。すべてのプロンプトに含める。要約しない

**「中間で見失う」効果**：
- モデルは長い入力の先頭と末尾を信頼性高く処理する
- 中間に埋もれた情報は見落とす可能性がある
- 修正：重要な結果サマリーを**先頭**に配置。明示的なセクションヘッダーを使用

**ツール結果のトリミング**：
- 注文検索が40以上のフィールドを返すが、必要なのは5つ
- コンテキストに追加する前に冗長な結果を関連フィールドに絞る
- 蓄積された無関係データによるトークン予算の枯渇を防止

**上流エージェントの最適化**：
- 冗長なコンテンツや推論チェーンの代わりに、構造化データ（キーファクト、引用、関連度スコア）を返すようエージェントを修正
- 下流エージェントのコンテキスト予算が限られている場合に重要

### タスクステートメント5.2：エスカレーションと曖昧性の解決

**3つの有効なエスカレーショントリガー**：
- **顧客が明示的に人間を要求**：即座に対応。先に解決を試みない
- **ポリシーの例外やギャップ**：文書化されたポリシー外のリクエスト
- **進行不能**：エージェントが解決を前進させられない

**2つの信頼性の低いトリガー**：
- **感情分析ベースのエスカレーション**：フラストレーションはケースの複雑さと相関しない
- **自己報告の信頼度スコア**：モデルは難しいケースで不正確に自信を持ち、簡単なケースで不確実になることが多い

**フラストレーションのニュアンス**：
- 問題が簡単で顧客がイライラしている場合：フラストレーションを認め、解決策を提示
- 顧客がヘルプ提供後に**再度**人間を要求した場合のみエスカレート
- ただし最初から「人間と話したい」と言った場合：調査なしで即エスカレート

### タスクステートメント5.3：エラー伝播

**構造化エラーコンテキスト**：
- 失敗タイプ（一時的、バリデーション、ビジネス、権限）
- 試行した内容（具体的なクエリ、使用パラメータ）
- 失敗前に収集された部分的結果
- 代替アプローチの可能性

**2つのアンチパターン**：
- **サイレント抑制**：空の結果を成功として返す → リカバリーを不可能にする
- **ワークフロー全体の終了**：単一障害でパイプライン全体を停止 → 部分的結果を破棄

**カバレッジアノテーション**：
- 合成出力はどの結果が十分に裏付けられ、どの領域にギャップがあるかを注記すべき
- 「ジャーナルアクセス不可のため地熱エネルギーのセクションは限定的」は黙って省略するより良い

### タスクステートメント5.4：コードベース探索

**コンテキスト劣化**：
- 長時間セッション：モデルが以前発見した具体的なクラスではなく「典型的なパターン」を参照し始める
- コンテキストが冗長な調査出力で埋まり、以前の発見を把握できなくなる

**緩和策**：
- **スクラッチパッドファイル**：重要な発見をファイルに書き出し、後続の質問で参照
- **サブエージェント委任**：具体的な調査にサブエージェントを生成、メインエージェントは高レベルの調整を維持
- **サマリー注入**：あるフェーズの発見を要約してから次のフェーズのサブエージェントを生成
- **`/compact`**：冗長な調査出力でコンテキストが埋まった時に使用量を削減

**クラッシュリカバリー**：
- 各エージェントが構造化状態を既知のファイル位置（マニフェスト）にエクスポート
- 再開時、コーディネーターがマニフェストをロードしてエージェントプロンプトに注入

### タスクステートメント5.5：人間のレビューと信頼度キャリブレーション

**集計メトリクスのトラップ**：
- 全体97%の精度でも、特定のドキュメントタイプで40%のエラー率が隠れている可能性
- 自動化前に常にドキュメントタイプ別・フィールドセグメント別で精度を検証

**層化ランダムサンプリング**：
- 高信頼度の抽出結果をサンプリングして継続検証
- 見逃されるはずの新しいエラーパターンを検出

**フィールドレベルの信頼度キャリブレーション**：
- モデルがフィールドごとに信頼度を出力
- ラベル付き検証セット（正解データ）で閾値をキャリブレーション
- 低信頼度フィールドを人間のレビューにルーティング
- 限られたレビュー能力を最も不確実な項目に優先配分

### タスクステートメント5.6：情報の出所

**構造化クレーム-ソースマッピング**：
- 各結果：主張 + ソースURL + ドキュメント名 + 関連抜粋 + 公開日
- 下流エージェントが合成を通じてこれらのマッピングを保持・統合
- これがないと、要約中に帰属情報が消失

**コンフリクト処理**：
- 2つの信頼できるソースが異なる統計を報告
- 一方を恣意的に選択**しない**
- 両方の値とソース帰属をアノテーション
- 消費者に判断を委ねる

**時間的認識**：
- 構造化出力に公開日/データ収集日を要求
- 異なる日付が異なる数値を説明する（矛盾ではない）

**コンテンツに適したレンダリング**：
- 財務データ → テーブル
- ニュース → 散文
- 技術的な結果 → 構造化リスト
- すべてを1つの均一なフォーマットに平坦化しない
