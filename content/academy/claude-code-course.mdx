# Claude Code in Action

[Claude Code in Action - Anthropic Academy](https://anthropic.skilljar.com/claude-code-in-action)

## コース概要

Claude Code in Action は、AI コーディングアシスタント「Claude Code」を開発ワークフローに統合するための**実践コース**です。Claude Code のアーキテクチャの理解から、実際の開発現場での活用、高度なインテグレーション戦略までを包括的にカバーします。

Claude Code は単なるチャットボットではなく、ファイルシステム、ターミナル、Git、外部サービスなどのツールを駆使してコードを理解・操作する**エージェント型コーディング環境**です。ユーザーがやりたいことを自然言語で記述すれば、Claude がコードベースを探索し、計画を立て、実装までを自律的に行います。

## 前提条件

- コマンドラインインターフェースとターミナル操作の基本的な知識
- Git を使ったバージョン管理の基本的な理解

## 対象者

- AI アシスタンスを開発ワークフローに統合したいソフトウェア開発者
- AI を活用した GitHub インテグレーションを導入したいチーム
- コーディングの生産性を向上させたいエンジニア

## 1. コーディングアシスタントのアーキテクチャ

AI システムがコードベースとどのように対話するかを理解します。ツール統合の仕組みと、コード分析を可能にする技術的基盤を学びます。

### Claude Code の動作環境

Claude Code はターミナル、VS Code、JetBrains IDE、デスクトップアプリ、ブラウザの各環境で利用できます。すべての環境は同一の Claude Code エンジンに接続されており、CLAUDE.md ファイル、設定、MCP サーバーはどの環境でも共有されます。

**インストール方法（ターミナル）:**

```bash
# macOS / Linux / WSL
curl -fsSL https://claude.ai/install.sh | bash

# Windows PowerShell
irm https://claude.ai/install.ps1 | iex

# Homebrew
brew install --cask claude-code
```

インストール後、任意のプロジェクトディレクトリで `claude` を実行して起動します。

### エージェントループの仕組み

Claude Code はエージェントループで動作します。ユーザーの指示を受け取ると、以下のサイクルを繰り返します:

1. **指示の解析** — ユーザーのプロンプトを理解する
2. **ツールの選択と実行** — ファイルの読み書き、コマンド実行、検索などのツールを使う
3. **結果の分析** — ツールの出力を分析し、次のアクションを決定する
4. **繰り返しまたは完了** — タスクが完了するまでこのサイクルを繰り返す

このアプローチにより、Claude Code は単一のファイル編集だけでなく、複数ファイルにまたがる複雑な変更、テストの実行と修正、Git 操作、プルリクエストの作成までを一連の流れとして実行できます。

### コンテキストウィンドウの管理

Claude のコンテキストウィンドウ（会話全体のメモリ）はセッション中の全メッセージ、読み取ったファイル内容、コマンド出力を保持します。**コンテキストが満杯に近づくと性能が低下する**ため、この管理が最も重要です。

- **`/clear`** — 無関係なタスク間でコンテキストをリセットする
- **`/compact <指示>`** — コンテキストを圧縮する（例: `/compact APIの変更に集中して`）
- **自動コンパクション** — コンテキスト限界に近づくと自動的に要約が行われる
- **サブエージェント** — 調査タスクを別コンテキストで実行し、メインの会話を汚さない

### できること一覧

| 機能 | 説明 |
|------|------|
| 機能開発・バグ修正 | 自然言語で記述するだけで、計画・実装・検証まで実行 |
| テスト作成 | テストされていないコードのテストを作成、実行、修正 |
| コミット・PR作成 | 変更のステージング、コミットメッセージ生成、ブランチ作成、PR作成 |
| コードレビュー | セキュリティ脆弱性、パフォーマンス問題、コード品質の分析 |
| リファクタリング | モダンなパターンへの移行、非推奨APIの更新 |
| ドキュメント生成 | JSDoc、docstring 等のコメント生成 |
| 外部ツール連携 | MCP経由でJira、Slack、Figma、DB等に接続 |
| マルチセッション | 複数のClaude Codeエージェントを並列実行 |

## 2. ツール利用システム

複数のツールを組み合わせて、複雑なマルチステップのプログラミングタスクに対応する方法を学びます。

### Claude Code が利用するツール

Claude Code は以下のカテゴリのツールを自動的に選択・組み合わせてタスクを遂行します:

**ファイル操作:**
- **Read** — ファイル内容の読み取り（画像やPDFも対応）
- **Write** — ファイルの新規作成または上書き
- **Edit** — ファイル内の特定箇所の精密な置換

**検索:**
- **Glob** — パターンマッチングによるファイル検索（例: `**/*.ts`）
- **Grep** — 正規表現によるファイル内容の検索

**コマンド実行:**
- **Bash** — シェルコマンドの実行（ビルド、テスト、Git操作等）

**対話:**
- **AskUserQuestion** — Plan Mode でユーザーに質問して要件を確認する

**拡張ツール:**
- **MCP ツール** — 外部サービスのツール（GitHub、Slack、DB等）
- **Skill** — カスタムスキルの呼び出し

### ツールのパーミッション管理

Claude Code はシステムを変更する可能性のあるアクション（ファイル書き込み、コマンド実行、MCPツール等）に対して、デフォルトでパーミッションを要求します。

**パーミッションを効率化する方法:**

1. **許可リスト** — 安全なコマンド（`npm run lint` や `git commit` 等）を事前に許可
2. **サンドボックス** — OS レベルの分離を有効にして、境界内で自由に作業させる
3. **`/permissions`** — インタラクティブにパーミッションルールを設定

```bash
# 完全にパーミッションをスキップ（サンドボックス環境専用）
claude --dangerously-skip-permissions
```

### マルチステップタスクの実行例

```bash
# テストの作成・実行・修正を一連で依頼
claude "write tests for the auth module, run them, and fix any failures"

# ビルドエラーの分析と修正
cat build-error.txt | claude -p "concisely explain the root cause of this build error"

# 変更ファイルのセキュリティレビュー
git diff main --name-only | claude -p "review these changed files for security issues"
```

### パイプライン統合

Claude Code は Unix 哲学に従い、パイプやスクリプトと組み合わせて使えます:

```bash
# ログ分析
tail -200 app.log | claude -p "Slack me if you see any anomalies"

# CI での翻訳自動化
claude -p "translate new strings into French and raise a PR for review"

# 構造化出力
claude -p "List all API endpoints" --output-format json

# ストリーミングJSON出力
claude -p "Analyze this log file" --output-format stream-json
```

## 3. コンテキスト管理（CLAUDE.md 等）

関連する会話コンテキストを維持し、プロジェクトリソースを効果的に参照する方法を学びます。

### CLAUDE.md と Auto Memory の2つのメモリシステム

Claude Code のセッションは毎回新しいコンテキストウィンドウで始まります。セッション間で知識を引き継ぐメカニズムは2つあります:

| | CLAUDE.md ファイル | Auto Memory |
|:--|:--|:--|
| **誰が書くか** | ユーザー | Claude 自身 |
| **内容** | 指示とルール | 学習したパターンと知見 |
| **スコープ** | プロジェクト、ユーザー、組織 | ワーキングツリー単位 |
| **読み込み** | 毎セッションの開始時（全文） | 毎セッションの開始時（先頭200行） |
| **用途** | コーディング規約、ワークフロー、アーキテクチャ | ビルドコマンド、デバッグの知見、発見した好み |

### CLAUDE.md の配置場所とスコープ

| スコープ | 配置場所 | 用途 | 共有範囲 |
|:--|:--|:--|:--|
| **管理ポリシー** | macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`<br/>Linux: `/etc/claude-code/CLAUDE.md` | 組織全体の指示 | 組織内全ユーザー |
| **プロジェクト** | `./CLAUDE.md` または `./.claude/CLAUDE.md` | チーム共有のプロジェクト指示 | ソース管理経由でチーム |
| **ユーザー** | `~/.claude/CLAUDE.md` | 個人的な全プロジェクト共通の好み | 自分のみ |

ワーキングディレクトリの上位階層にある CLAUDE.md はすべて起動時に読み込まれます。サブディレクトリの CLAUDE.md は、Claude がそのディレクトリのファイルを読む時にオンデマンドで読み込まれます。

### 効果的な CLAUDE.md の書き方

**サイズ:** 1ファイルあたり200行以下を目標にします。長いファイルはコンテキストを消費し、遵守率を下げます。

**構造:** Markdown のヘッダーと箇条書きで関連する指示をグループ化します。

**具体性:** 検証可能な具体的な指示を書きます。

```markdown
# コードスタイル
- ES modules (import/export) 構文を使う。CommonJS (require) は使わない
- 可能な場合はインポートを分割構造化する (例: import { foo } from 'bar')

# ワークフロー
- コード変更の後は必ず型チェックを実行する
- パフォーマンスのため、テストスイート全体ではなく単一テストを優先する
```

**含めるべき内容:**
- Claude が推測できないBashコマンド
- デフォルトと異なるコードスタイルルール
- テスト指示とテストランナーの指定
- リポジトリの作法（ブランチ命名、PRの慣例）
- アーキテクチャの決定事項
- 環境変数の必要条件

**含めるべきでない内容:**
- Claude がコードを読めばわかること
- 標準的な言語慣例
- 頻繁に変わる情報
- 長い説明やチュートリアル

### `/init` による自動生成

```bash
claude
# セッション内で:
/init
```

`/init` はコードベースを分析して、ビルドコマンド、テスト指示、プロジェクト慣例を含む初期 CLAUDE.md を生成します。既存の CLAUDE.md がある場合は改善提案を行います。

### ファイルインポート

CLAUDE.md は `@path/to/import` 構文で追加ファイルをインポートできます:

```markdown
# プロジェクト概要は @README.md を参照
# 利用可能なnpmコマンドは @package.json を参照

# 追加指示
- Gitワークフロー: @docs/git-instructions.md
- 個人設定: @~/.claude/my-project-instructions.md
```

### `.claude/rules/` によるルールの分割管理

大規模プロジェクトでは、指示を複数のファイルに分割して管理できます:

```
your-project/
├── .claude/
│   ├── CLAUDE.md           # メインのプロジェクト指示
│   └── rules/
│       ├── code-style.md   # コードスタイルガイドライン
│       ├── testing.md      # テスト慣例
│       └── security.md     # セキュリティ要件
```

**パス指定ルール** — 特定のファイルに対してのみ適用されるルールを YAML フロントマターで定義:

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 開発ルール

- すべてのAPIエンドポイントに入力バリデーションを含める
- 標準エラーレスポンス形式を使用する
- OpenAPI ドキュメンテーションコメントを含める
```

### Auto Memory の仕組み

Auto Memory は Claude がセッション中に自動的に学習内容をメモする機能です。ビルドコマンド、デバッグの知見、アーキテクチャノート、コードスタイルの好み、ワークフローの習慣が保存されます。

保存先: `~/.claude/projects/<project>/memory/`

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # インデックス（毎セッション読み込み、先頭200行）
├── debugging.md       # デバッグパターンの詳細
├── api-conventions.md # API設計の決定事項
└── ...                # Claudeが作成するトピックファイル
```

`/memory` コマンドで、読み込まれているすべての CLAUDE.md とルールファイルを一覧表示し、Auto Memory のオン/オフを切り替えたり、メモリファイルを編集したりできます。

## 4. ビジュアルコミュニケーションワークフロー

ビジュアル入力を使ってインターフェースの変更を伝える方法と、コードベース変更のためのプランニング機能を活用する方法を学びます。

### 画像を使った作業

Claude Code は画像を直接受け取って分析できます。以下の方法で画像を提供できます:

1. **ドラッグ&ドロップ** — Claude Code ウィンドウに画像をドラッグ
2. **ペースト** — 画像をコピーし、`Ctrl+V` で貼り付け（macOS で `Cmd+V` ではなく `Ctrl+V`）
3. **ファイルパス指定** — `"Analyze this image: /path/to/image.png"` と指定

**活用例:**

```
# スクリーンショットからのエラー診断
Here's a screenshot of the error. What's causing it?

# デザインモックアップからのCSS生成
Generate CSS to match this design mockup

# UIコンポーネントの再現
What HTML structure would recreate this component?

# データベーススキーマ図の分析
This is our current database schema. How should we modify it for the new feature?
```

**ベストプラクティス:**

テキストだけでは説明しにくい場合に画像を使います。特に UI 変更では「期待するデザインのスクリーンショットを貼って指示し、結果のスクリーンショットを撮って比較させ、差異をリストアップして修正させる」というパターンが効果的です。

### Plan Mode（計画モード）

Plan Mode は Claude にコードベースの読み取り専用分析を行わせ、実装計画を作成させるモードです。コード探索、複雑な変更の計画、コードレビューに最適です。

**起動方法:**

```bash
# コマンドラインから Plan Mode で開始
claude --permission-mode plan

# ヘッドレスモードで Plan Mode 実行
claude --permission-mode plan -p "Analyze the authentication system and suggest improvements"
```

**セッション中の切り替え:** `Shift+Tab` でモードを切り替えます。

**推奨ワークフロー:**

1. **探索** — Plan Mode で入り、コードを読んで理解する
2. **計画** — 詳細な実装計画を作成させる（`Ctrl+G` でエディタで編集可能）
3. **実装** — Normal Mode に戻し、計画に基づいてコーディング
4. **コミット** — 変更をコミットし、PRを作成

```
# Plan Mode での例
I need to refactor our authentication system to use OAuth2.
Create a detailed migration plan.

# フォローアップ
What about backward compatibility?
How should we handle database migration?
```

Plan Mode をデフォルトにする設定:

```json
// .claude/settings.json
{
  "permissions": {
    "defaultMode": "plan"
  }
}
```

### @ によるファイル・ディレクトリ参照

`@` を使ってファイルやディレクトリを即座にコンテキストに追加できます:

```
# ファイル参照
Explain the logic in @src/utils/auth.js

# ディレクトリ参照
What's the structure of @src/components?

# MCP リソース参照
Show me the data from @github:repos/owner/repo/issues

# 複数ファイル参照
Compare @file1.js and @file2.js
```

## 5. カスタムオートメーション（Skills, Hooks, Commands）

繰り返し発生する開発タスクを効率化するための、再利用可能なコマンドとオートメーションの構築方法を学びます。

### Skills（スキル）

Skills は Claude の能力を拡張する再利用可能な指示セットです。`SKILL.md` ファイルに指示を書くと、Claude は関連するタスクで自動的にスキルを適用するか、`/skill-name` で直接呼び出せます。

**スキルの作成:**

```bash
mkdir -p ~/.claude/skills/explain-code
```

`~/.claude/skills/explain-code/SKILL.md`:

```yaml
---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when explaining how code works.
---

When explaining code, always include:

1. **Start with an analogy**: Compare the code to something from everyday life
2. **Draw a diagram**: Use ASCII art to show the flow
3. **Walk through the code**: Explain step-by-step
4. **Highlight a gotcha**: Common mistake or misconception
```

**スキルの配置場所:**

| 場所 | パス | 適用範囲 |
|:--|:--|:--|
| エンタープライズ | 管理設定で指定 | 組織内全ユーザー |
| パーソナル | `~/.claude/skills/<name>/SKILL.md` | 全プロジェクト |
| プロジェクト | `.claude/skills/<name>/SKILL.md` | このプロジェクトのみ |

**フロントマターオプション:**

| フィールド | 説明 |
|:--|:--|
| `name` | スキルの表示名（省略時はディレクトリ名） |
| `description` | スキルの説明と使用条件。Claude が自動適用を判断する |
| `disable-model-invocation` | `true` にすると Claude は自動呼び出し不可（手動 `/name` 専用） |
| `user-invocable` | `false` にすると `/` メニュー非表示（バックグラウンド知識用） |
| `allowed-tools` | スキル使用時に許可されるツール |
| `context` | `fork` でサブエージェントとして実行 |
| `agent` | `context: fork` 時のサブエージェントタイプ |

**引数の受け渡し:**

```yaml
---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---

Fix GitHub issue $ARGUMENTS following our coding standards.
1. Read the issue description
2. Implement the fix
3. Write tests
4. Create a commit
```

`/fix-issue 123` で呼び出すと、`$ARGUMENTS` が `123` に置換されます。

**動的コンテキスト注入:**

```yaml
---
name: pr-summary
description: Summarize changes in a pull request
context: fork
agent: Explore
---

## Pull request context
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`

## Your task
Summarize this pull request...
```

`` !`command` `` 構文でシェルコマンドを事前実行し、その出力をスキル内容に挿入できます。

**バンドルスキル（組み込み）:**

| スキル | 用途 |
|:--|:--|
| `/batch <instruction>` | コードベース全体にわたる大規模な並列変更 |
| `/claude-api` | Claude API リファレンスの読み込み |
| `/debug [description]` | セッションデバッグログのトラブルシューティング |
| `/loop [interval] <prompt>` | プロンプトを定期的に繰り返し実行 |
| `/simplify [focus]` | 変更したファイルのコード品質レビューと修正 |

### Hooks（フック）

Hooks は Claude Code のライフサイクルの特定のポイントで自動的に実行されるアクションです。CLAUDE.md の指示が「アドバイス」であるのに対し、Hooks は**確定的に実行される保証**があります。

**フックハンドラの4タイプ:**

1. **command** — シェルコマンドを実行
2. **http** — HTTP POST リクエストを送信
3. **prompt** — Claude モデルに yes/no の判断を求める
4. **agent** — ツールを使えるサブエージェントを起動

**主要なフックイベント:**

| カテゴリ | イベント | タイミング | ブロック可能 |
|:--|:--|:--|:--|
| セッション | `SessionStart` | セッション開始時 | 不可 |
| セッション | `SessionEnd` | セッション終了時 | 不可 |
| ツール | `PreToolUse` | ツール実行前 | 可（ツール呼び出しをブロック） |
| ツール | `PostToolUse` | ツール成功後 | 不可 |
| 入力 | `UserPromptSubmit` | プロンプト送信時 | 可（プロンプトをブロック） |
| 停止 | `Stop` | Claude が応答完了時 | 可（継続を強制） |
| 設定 | `ConfigChange` | 設定ファイル変更時 | 可 |

**設定例（ファイル編集後に自動フォーマット）:**

```json
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/check-style.sh"
          }
        ]
      }
    ]
  }
}
```

**破壊的コマンドのブロック例:**

```bash
#!/bin/bash
COMMAND=$(jq -r '.tool_input.command')

      permissionDecision: "deny",
      permissionDecisionReason: "Destructive command blocked"
    }
  }'
else
  exit 0
fi
```

## 6. MCP サーバー統合

外部ツールやサービスを統合して、Claude Code の機能を拡張する方法を学びます。

### MCP（Model Context Protocol）とは

MCP は AI ツールと外部データソースを接続するためのオープンスタンダードです。MCP サーバーを通じて、Claude Code はイシュートラッカー、データベース、モニタリングツール、デザインツールなど、あらゆる外部サービスと連携できます。

**MCP で可能になること:**

- イシュートラッカーから機能を実装し、PRを作成
- モニタリングデータを分析（Sentry、Statsig等）
- データベースにクエリを実行
- Figma のデザインを統合
- メール下書きの自動作成
- Telegram、Discord 等の外部イベントに反応

### MCP サーバーのインストール

**リモート HTTP サーバー（推奨）:**

```bash
# 基本構文
claude mcp add --transport http <name> <url>

# Notion への接続
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Bearer トークン付き
claude mcp add --transport http secure-api https://api.example.com/mcp \
  --header "Authorization: Bearer your-token"
```

**リモート SSE サーバー:**

```bash
claude mcp add --transport sse asana https://mcp.asana.com/sse
```

**ローカル stdio サーバー:**

```bash
claude mcp add --transport stdio --env AIRTABLE_API_KEY=YOUR_KEY airtable \
  -- npx -y airtable-mcp-server
```

### MCP サーバーの管理

```bash
# 一覧表示
claude mcp list

# 詳細確認
claude mcp get github

# 削除
claude mcp remove github

# セッション内でステータス確認
/mcp
```

### スコープ設定

| スコープ | 保存先 | 用途 |
|:--|:--|:--|
| **local**（デフォルト） | `~/.claude.json` | 個人用、現在のプロジェクトのみ |
| **project** | `.mcp.json`（リポジトリ共有） | チーム共有のツール |
| **user** | `~/.claude.json` | 全プロジェクト共通の個人ツール |

```bash
# プロジェクトスコープで追加（チーム共有）
claude mcp add --transport http paypal --scope project https://mcp.paypal.com/mcp
```

### 実践例: Sentry でのエラーモニタリング

```bash
# Sentry MCP サーバーを追加
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# 認証（セッション内で）
/mcp
```

認証後:

```
What are the most common errors in the last 24 hours?
Show me the stack trace for error ID abc123
Which deployment introduced these new errors?
```

### チャンネル（プッシュメッセージ）

MCP サーバーは `claude/channel` 機能を宣言することで、Claude のセッションにプッシュメッセージを送信できます。CI 結果、モニタリングアラート、チャットメッセージなど外部イベントに Claude がリアクティブに対応できます。

### `.mcp.json` での環境変数展開

```json
{
  "mcpServers": {
    "api-server": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": {
        "Authorization": "Bearer ${API_KEY}"
      }
    }
  }
}
```

`${VAR}` または `${VAR:-default}` 構文で環境変数を展開できます。

## 7. GitHub インテグレーション

自動化されたコードレビュープロセスのセットアップと、バージョン管理ワークフローへの AI アシスタンスの統合方法を学びます。

### Claude Code GitHub Actions

GitHub Actions で Claude Code を統合すると、PR やイシューのコメントで `@claude` とメンションするだけで、コード分析、PR作成、機能実装、バグ修正が可能になります。

**主な機能:**

- **即時PR作成** — 必要な内容を記述するだけで完全なPRを作成
- **自動コード実装** — イシューからコードを生成
- **プロジェクト標準の遵守** — `CLAUDE.md` のガイドラインに従う
- **セキュアなデフォルト** — コードはGitHubのランナー上で処理

### セットアップ方法

**クイックセットアップ（推奨）:**

```bash
claude
# セッション内で:
/install-github-app
```

**手動セットアップ:**

1. [Claude GitHub App](https://github.com/apps/claude) をリポジトリにインストール
2. `ANTHROPIC_API_KEY` をリポジトリシークレットに追加
3. ワークフローファイルを `.github/workflows/` にコピー

### 基本ワークフロー

```yaml
name: Claude Code
on:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]
jobs:
  claude:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          # @claude メンションに応答
```

### PRでのコードレビュー

```yaml
name: Code Review
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: "Review this pull request for code quality, correctness, and security."
          claude_args: "--max-turns 5"
```

### スケジュール実行

```yaml
name: Daily Report
on:
  schedule:
    - cron: "0 9 * * *"
jobs:
  report:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: "Generate a summary of yesterday's commits and open issues"
          claude_args: "--model opus"
```

### 一般的なコマンド例

イシューやPRのコメントで:

```
@claude implement this feature based on the issue description
@claude how should I implement user authentication for this endpoint?
@claude fix the TypeError in the user dashboard component
```

### アクションパラメータ

| パラメータ | 説明 | 必須 |
|:--|:--|:--|
| `prompt` | Claude への指示 | 任意 |
| `claude_args` | CLI 引数 | 任意 |
| `anthropic_api_key` | API キー | はい（直接API利用時） |
| `github_token` | GitHub トークン | 任意 |
| `trigger_phrase` | トリガーフレーズ（デフォルト: `@claude`） | 任意 |
| `use_bedrock` | AWS Bedrock 利用 | 任意 |
| `use_vertex` | Google Vertex AI 利用 | 任意 |


Claude Code で PR を作成すると、セッションが自動的にその PR にリンクされます。後で `claude --from-pr <number>` で再開できます。

## 8. 推論モード（thinking, planning）

さまざまな複雑さのプログラミング課題に対して、thinking（思考）モードや planning（計画）モードなどの推論アプローチを適用する方法を学びます。

### Extended Thinking（拡張思考）

Extended Thinking はデフォルトで有効であり、Claude が応答する前にステップバイステップで複雑な問題を推論するための時間を確保します。verbose モード（`Ctrl+O`）で思考プロセスをグレーのイタリック体テキストとして確認できます。

**特に効果的な場面:**
- 複雑なアーキテクチャの決定
- 困難なバグの調査
- マルチステップの実装計画
- 異なるアプローチ間のトレードオフ評価

### Adaptive Reasoning（適応的推論）

Opus 4.6 および Sonnet 4.6 は adaptive reasoning をサポートしています。固定の思考トークン予算ではなく、エフォートレベルの設定に基づいてモデルが動的に思考量を配分します。

### 思考モードの設定方法

| 設定方法 | 操作 | 詳細 |
|:--|:--|:--|
| **エフォートレベル** | `/effort` コマンド、`/model` 内で調整、`CLAUDE_CODE_EFFORT_LEVEL` 環境変数 | Opus 4.6 / Sonnet 4.6 での思考の深さを制御 |
| **`ultrathink` キーワード** | プロンプトに "ultrathink" を含める | そのターンのみエフォートを高に設定 |
| **トグルショートカット** | macOS: `Option+T` / Windows・Linux: `Alt+T` | 現在のセッションで思考のオン/オフ |
| **グローバルデフォルト** | `/config` で思考モードを切り替え | `~/.claude/settings.json` の `alwaysThinkingEnabled` |
| **トークン予算の制限** | `MAX_THINKING_TOKENS` 環境変数 | 思考トークン数を制限 |

### Extended Thinking の仕組み

**Opus 4.6 / Sonnet 4.6 の場合:**
- Adaptive reasoning により、エフォートレベルに基づいて思考トークンが動的に配分される
- `MAX_THINKING_TOKENS` は `0`（無効化）に設定した場合のみ適用
- `CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1` で固定予算に戻せる

**旧モデルの場合:**
- 固定トークン予算が出力割り当てから使用される
- `MAX_THINKING_TOKENS` で予算を制限可能
- `/config` や `Option+T`/`Alt+T` で無効化可能

### 推論モードの使い分け

| タスクの複雑さ | 推奨アプローチ |
|:--|:--|
| 単純（タイポ修正、ログ追加、変数名変更） | 直接実行。Plan Mode やプランニング不要 |
| 中程度（単一ファイルの変更、明確なバグ修正） | Extended Thinking をデフォルトで利用 |
| 複雑（複数ファイルの変更、アーキテクチャ変更） | Plan Mode で探索→計画→実装 |
| 非常に複雑（大規模リファクタリング、新機能実装） | Plan Mode + `ultrathink` + サブエージェント活用 |

## Enterprise セットアップ

組織で Claude Code を導入する場合、Amazon Bedrock や Google Cloud Vertex AI を通じたエンタープライズセットアップが可能です。

### AWS Bedrock での利用

**前提条件:**
- Amazon Bedrock で Claude モデルへのアクセスを有効化
- GitHub OIDC Identity Provider を AWS に設定
- Bedrock 権限を持つ IAM ロールを作成

```yaml
# GitHub Actions ワークフロー（抜粋）
- name: Configure AWS Credentials (OIDC)
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: ${{ secrets.AWS_ROLE_TO_ASSUME }}
    aws-region: us-west-2

- uses: anthropics/claude-code-action@v1
  with:
    github_token: ${{ steps.app-token.outputs.token }}
    use_bedrock: "true"
    claude_args: '--model us.anthropic.claude-sonnet-4-6 --max-turns 10'
```

### Google Cloud Vertex AI での利用

**前提条件:**
- Vertex AI API を Google Cloud プロジェクトで有効化
- Workload Identity Federation を GitHub Actions 用に設定
- Vertex AI 権限を持つサービスアカウントを作成

```yaml
# GitHub Actions ワークフロー（抜粋）
- name: Authenticate to Google Cloud
  uses: google-github-actions/auth@v2
  with:
    workload_identity_provider: ${{ secrets.GCP_WORKLOAD_IDENTITY_PROVIDER }}
    service_account: ${{ secrets.GCP_SERVICE_ACCOUNT }}

- uses: anthropics/claude-code-action@v1
  with:
    github_token: ${{ steps.app-token.outputs.token }}
    use_vertex: "true"
    claude_args: '--model claude-sonnet-4@20250514 --max-turns 10'
  env:
    ANTHROPIC_VERTEX_PROJECT_ID: ${{ steps.auth.outputs.project_id }}
    CLOUD_ML_REGION: us-east5
```

### 組織全体の管理設定

管理ポリシーで組織全体に適用される CLAUDE.md を配布できます:

- macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`
- Linux/WSL: `/etc/claude-code/CLAUDE.md`
- Windows: `C:\Program Files\ClaudeCode\CLAUDE.md`

MDM、Group Policy、Ansible 等の構成管理ツールで開発者マシン全体に配布します。管理ポリシーの CLAUDE.md は個人の設定で除外できません。

| 関心事 | 設定先 |
|:--|:--|
| ツール、コマンド、ファイルパスのブロック | 管理設定: `permissions.deny` |
| サンドボックス分離の強制 | 管理設定: `sandbox.enabled` |
| 環境変数とAPIプロバイダルーティング | 管理設定: `env` |
| 認証方法と組織ロック | 管理設定: `forceLoginMethod`, `forceLoginOrgUUID` |
| コードスタイルとガイドライン | 管理 CLAUDE.md |
| データ取り扱いとコンプライアンス | 管理 CLAUDE.md |

## ベストプラクティスまとめ

### 最も重要な原則

1. **検証手段を提供する** — テスト、スクリーンショット、期待出力を含めて Claude が自己検証できるようにする。これが最もレバレッジの高い施策。

2. **探索→計画→実装の順序** — いきなりコーディングさせず、Plan Mode で調査と計画を分離する。

3. **具体的なコンテキストを提供する** — ファイルを特定し、制約を述べ、パターンを示す。

4. **コンテキストを積極的に管理する** — `/clear` でリセット、`/compact` で圧縮、サブエージェントで調査を分離。

5. **早期に軌道修正する** — `Esc` で停止、`Esc+Esc` で巻き戻し、`/rewind` でチェックポイントに戻す。

### 避けるべきパターン

| パターン | 問題 | 対策 |
|:--|:--|:--|
| キッチンシンクセッション | 無関係なタスクが混在しコンテキストが汚染 | タスク間で `/clear` |
| 繰り返しの修正 | 失敗アプローチでコンテキストが汚染 | 2回失敗したら `/clear` して最初からやり直す |
| 巨大な CLAUDE.md | 長すぎて重要なルールが無視される | 定期的に刈り込む |
| 検証なしの信頼 | エッジケースを見落とした実装 | 必ず検証手段を提供 |
| 無限の探索 | スコープ不定の調査でコンテキストが満杯 | 調査はサブエージェントに委任 |

### セッション管理テクニック

```bash
# 最後のセッションを再開
claude --continue

# セッションを名前で管理
claude -n auth-refactor

# 名前でセッションを再開
claude --resume auth-refactor

# セッションピッカーを開く
claude --resume

# PRにリンクされたセッションを再開
claude --from-pr 123
```

### 並列実行パターン

**Writer/Reviewer パターン:**

| セッション A（Writer） | セッション B（Reviewer） |
|:--|:--|
| `Implement a rate limiter` | |
| | `Review the rate limiter implementation. Look for edge cases.` |
| `Address the review feedback: [Session B output]` | |
