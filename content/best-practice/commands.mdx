# コマンド設計パターン

[Commands - Claude Code Best Practice](https://github.com/shanraisshan/claude-code-best-practice)

## コマンドとは

コマンドは、**既存のコンテキストに注入される知識**です。サブエージェントとは異なり、新しいコンテキストを生成せず、現在のセッション内で動作するプロンプトテンプレートです。ユーザーがスラッシュ（`/`）で呼び出すワークフローオーケストレーション用のツールです。

**配置場所:** `.claude/commands/<name>.md`

```
.claude/
└── commands/
    ├── review.md
    ├── test.md
    ├── deploy-check.md
    └── refactor/
        ├── extract-function.md
        └── rename-variable.md
```

コマンドファイルは Git に管理され、チーム全体で共有できます。

## コマンドとサブエージェントの使い分け

Boris Cherny（Claude Code 開発者）の推奨：

> 「ワークフローにはサブエージェントではなくコマンドを使いましょう」

| 特徴 | コマンド | サブエージェント |
|------|----------|------------------|
| コンテキスト | 既存セッション内 | 独立した新規セッション |
| 用途 | 繰り返しのワークフロー | 複雑な独立タスク |
| オーバーヘッド | 低い | 高い |
| 状態共有 | 現在の状態を引き継ぐ | 分離された状態 |

## 設計原則

### 1日に複数回行う作業をコマンド化する

> 「1日に1回以上行う作業は、スキルかコマンドにしましょう」

毎日の内部ループで何度も実行するワークフローをコマンドにすると、繰り返しのプロンプト入力を省けます。

### コマンドファイルの書き方

```markdown
<!-- .claude/commands/review.md -->
# コードレビュー

以下の手順でコードレビューを実行してください：

1. `git diff main...HEAD` で変更差分を取得
2. 変更されたファイルごとに以下を確認：
   - 型安全性
   - エラーハンドリング
   - テストカバレッジ
3. 問題点をリスト形式で報告
4. 改善提案をコード例付きで提示

## 出力フォーマット
- 🔴 重大: 即座に修正が必要
- 🟡 警告: 改善推奨
- 🟢 良好: 問題なし
```

### 引数の活用

コマンドには `$ARGUMENTS` プレースホルダーで引数を渡せます。

```markdown
<!-- .claude/commands/test.md -->
# テスト実行

$ARGUMENTS で指定されたモジュールのテストを実行してください。

1. 対象モジュール: $ARGUMENTS
2. ユニットテストを実行
3. 失敗したテストの原因を分析
4. 修正案を提示
```

呼び出し例：
```
/test auth-module
```

## オーケストレーションパターン

### Command → Agent → Skill パターン

コマンドがエントリポイントとなり、必要に応じてエージェントやスキルを呼び出す階層的パターンです。

```
/weather-orchestrator
  ├── Agent: data-collector（データ収集）
  │   └── Skill: api-fetcher
  ├── Agent: analyzer（分析）
  │   └── Skill: data-processor
  └── 最終レポート生成
```

### ディレクトリ構造での整理

サブディレクトリを使ってコマンドを分類できます。

```
.claude/commands/
├── dev/
│   ├── test.md
│   ├── lint.md
│   └── build.md
├── review/
│   ├── security.md
│   ├── performance.md
│   └── accessibility.md
└── deploy/
    ├── staging.md
    └── production.md
```

## 実践的なコマンド例

### コミット用コマンド

```markdown
<!-- .claude/commands/commit.md -->
# スマートコミット

1. `git status` と `git diff --staged` で変更を確認
2. 変更内容を分析し、Conventional Commits 形式のメッセージを生成
3. 変更が大きい場合は複数コミットに分割を提案
4. ユーザーの確認後にコミット実行
```

### デバッグ用コマンド

```markdown
<!-- .claude/commands/debug.md -->
# デバッグアシスタント

$ARGUMENTS のエラーを調査してください。

1. エラーメッセージを解析
2. 関連するソースコードを特定
3. スタックトレースを追跡
4. 根本原因を特定
5. 修正案をコード付きで提示
```

## ベストプラクティスまとめ

| 項目 | 推奨 |
|------|------|
| コマンド化の基準 | 1日に複数回行う作業 |
| サブエージェントとの使い分け | ワークフローにはコマンドを優先 |
| ファイル管理 | Git管理でチーム共有 |
| 構造化 | サブディレクトリで分類 |
| 引数 | `$ARGUMENTS` で柔軟に |
| オーケストレーション | Command → Agent → Skill の階層パターン |
