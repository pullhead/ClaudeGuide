# Tips & Tricks（75+）

[Tips & Tricks - Claude Code Best Practice](https://github.com/shanraisshan/claude-code-best-practice)

Claude Code の実践的なテクニック集です。開発者コミュニティと Claude Code チームの知見をカテゴリ別にまとめています。

## プロンプティング（3 Tips）

1. **検証を要求する** — Claude の最初の解決策をそのまま受け入れず、「本当に正しいか検証して」と確認を促す
2. **エレガントな代替案を求める** — 微妙な修正には「もっと良い方法を考えて」と再考を促す
3. **自律的に解決させる** — バグを貼り付けて「fix」と言うだけで十分。アプローチをマイクロマネジメントしない

## 計画・仕様（6 Tips）

4. **プランモードから始める** — 実装前に必ず計画を立てる
5. **AskUserQuestion ツールを活用** — Claude にインタビューさせ、その結果をフレッシュセッションで実行する
6. **フェーズゲート付き計画** — 各フェーズに複数種類のテスト（ユニット、自動化、統合）を設ける
7. **セカンドオピニオン** — 別の Claude インスタンスでスタッフエンジニア視点のレビューを実施
8. **詳細な仕様を書く** — 曖昧さを最小化してからタスクを渡す
9. **プロトタイプ重視** — 長大なPRDを書くより、20〜30のプロトタイプを作る方がコスト効率が良い

## CLAUDE.md（7 Tips）

10. **200行以内ルール** — 1ファイルあたり200行以内、理想は60行程度
11. **条件付きタグ** — `<important if="...">` でドメイン固有ルールを囲み、無視されるのを防ぐ
12. **モノレポ対応** — 複数の CLAUDE.md を配置し、祖先・子孫の自動ロードを活用
13. **ルール分割** — `.claude/rules/` ディレクトリに大きな指示を分割
14. **実行可能なセットアップ** — memory.md や constitution.md より、すぐ動くテスト環境を重視
15. **テスト即実行** — どの開発者でも Claude 起動直後にテスト実行可能な状態を維持
16. **マイグレーション完了** — 部分的なマイグレーションはモデルを混乱させる。完了させてから作業する

## エージェント（4 Tips）

17. **機能特化型** — 汎用ロール（QA、バックエンド）ではなく、機能特化型サブエージェントを作る
18. **計算負荷の分散** — サブエージェントでタスクをオフロードし、メインコンテキストの焦点を維持
19. **エージェントチーム** — tmux + git worktree で並列開発
20. **テスト時間計算量** — 別コンテキストウィンドウがバグ検出精度を向上させる

## コマンド（3 Tips）

21. **コマンドを優先** — ワークフロー自動化にはサブエージェントよりコマンドを使う
22. **繰り返し作業をコマンド化** — 1日に複数回行う内部ループのタスクをスラッシュコマンドにする
23. **スキルかコマンドに変換** — 頻繁に繰り返すアクションは効率化のためにコマンド化する

## スキル（9 Tips）

24. **コンテキストフォーク** — `context: fork` でサブエージェント実行し、メインは結果のみ受け取る
25. **モノレポ整理** — サブフォルダでスキルを構造化する
26. **フォルダ構成** — references/、scripts/、examples/ サブディレクトリで段階的開示
27. **Gotchas セクション** — 失敗パターンとエッジケースを文書化する
28. **トリガー記述** — 要約ではなく「いつ発動すべきか？」の観点で記述
29. **自明なことを書かない** — デフォルト動作を超えた振る舞いにフォーカス
30. **ゴールと制約** — 手順の処方箋ではなく、目標と制約条件を提供する
31. **合成可能な素材** — スクリプトやライブラリを同梱し、ゼロからの再構築を避ける
32. **動的インジェクション** — `` !`command` `` 構文でシェル出力を実行時に注入

## フック（5 Tips）

33. **オンデマンドフック** — `/careful` で破壊的コマンドをブロック
34. **利用状況計測** — PreToolUse フックでスキルの発火パターンを追跡
35. **自動フォーマット** — PostToolUse フックで CI 失敗を事前回避
36. **パーミッションルーティング** — Opus にルーティングして攻撃スキャン＋自動承認
37. **Stop フック** — ターン完了時に検証や継続を促す

## ワークフロー（8 Tips）

38. **ダムゾーン回避** — コンテキスト使用量 50% で手動 `/compact` を実行
39. **素のClaude が勝つ場面** — 小さなタスクには複雑なワークフローは不要
40. **モデルの使い分け** — Opus で計画、Sonnet でコーディング
41. **思考モード有効化** — Explanatory 出力スタイルで意思決定の可視性を向上
42. **ultrathink** — 高負荷推論が必要な場面でキーワードとして使用
43. **セッション管理** — 重要なセッションに名前を付けて再開可能にする
44. **/rewind を活用** — 同じコンテキストで修正するより Esc Esc で取り消す方が効率的
45. **頻繁なコミット** — タスク完了時に最低1時間に1回コミット

## 高度なワークフロー（7 Tips）

46. **ASCII ダイアグラム** — アーキテクチャ理解にASCII図を多用する
47. **`/loop` 活用** — 最大3日間の定期モニタリングタスクに使用
48. **Ralph Wiggum プラグイン** — 長時間の自律開発タスク向け
49. **ワイルドカードパーミッション** — `dangerously-skip-permissions` の代わりに使用
50. **`/sandbox` 活用** — パーミッションプロンプトを84%削減
51. **@claude タグ** — 同僚の PR に @claude をタグ付けしてリントルールを自動生成
52. **プロダクト検証スキル** — signup-flow-driver、checkout-verifier 等に投資する

## デバッグ（7 Tips）

53. **スクリーンショット共有** — デバッグが複雑になったらスクリーンショットを Claude に共有
54. **ブラウザMCP活用** — Chrome、Playwright、Chrome DevTools MCP でコンソールログにアクセス
55. **バックグラウンド実行** — ターミナルコマンドをバックグラウンドタスクとして実行しログ可視性を向上
56. **`/doctor` コマンド** — インストールや設定の問題を診断
57. **コンパクションエラー対策** — コンパクト前に1Mトークンモデルを選択
58. **クロスモデルQA** — Codex など他のモデルインスタンスでQAとレビュー
59. **エージェント検索を優先** — RAGよりも glob + grep のエージェント検索がコードドリフトに強い

## ユーティリティ（5 Tips）

60. **ターミナルエミュレータ** — IDE環境ではなく iTerm、Ghostty、tmux を使用
61. **Wispr Flow** — 音声入力で生産性を向上
62. **claude-code-hooks** — フィードバック機構をデプロイ
63. **ステータスライン** — コンテキスト認識のためのステータスラインツールを有効化
64. **Plans Directory** — settings.json の Plans Directory 等の機能を探索

## 日常習慣（3 Tips）

65. **毎日アップデート** — Claude Code を毎日更新し、changelog で改善点を確認
66. **コミュニティ監視** — r/ClaudeAI、r/ClaudeCode をモニタリング
67. **開発者をフォロー** — Claude Code チームメンバーの SNS をフォローして最新情報をキャッチ

## 追加のプロ Tips

68. **「サブエージェントを使って」** — 複雑な問題に計算リソースを追加投入する魔法の言葉
69. **フレッシュセッションの活用** — コンテキストが汚染されたら新しいセッションで再開
70. **段階的な複雑化** — シンプルなセットアップから始め、必要に応じて拡張
71. **テストを最初に書く** — Claude にテストファーストで実装させると品質が向上
72. **既存パターンの参照** — 新しいコードを書く前に、既存コードのパターンを Claude に学習させる
73. **1つずつ** — 一度に複数の大きなタスクを頼まず、1つずつ着実に進める
74. **コンテキストの質** — 大量の情報より、高品質で関連性の高い情報をコンテキストに入れる
75. **失敗から学ぶ** — うまくいかなかったパターンを Gotchas セクションに記録して蓄積する
