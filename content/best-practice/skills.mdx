# スキルモジュール設計

[Skills - Claude Code Best Practice](https://github.com/shanraisshan/claude-code-best-practice)

## スキルとは

スキルは、**自動検出・プリロード可能な知識モジュール**です。コマンドと同様に既存コンテキストに注入されますが、段階的開示（progressive disclosure）とコンテキストフォークに対応している点が異なります。

**配置場所:** `.claude/skills/<name>/SKILL.md`

スキルは**ファイルではなくフォルダ**として構成されます。

```
.claude/skills/
└── security-review/
    ├── SKILL.md           # スキル定義
    ├── references/        # 参照資料
    │   └── owasp-top10.md
    ├── scripts/           # ヘルパースクリプト
    │   └── scan.sh
    └── examples/          # 使用例
        └── auth-review.md
```

## SKILL.md の書き方

### トリガー記述を最適化する

スキルの説明は**人間向けの要約ではなく、モデル向けのトリガー記述**として書きます。「このスキルがいつ発動すべきか？」という観点で記述します。

```markdown
# Security Review Skill

## トリガー
認証、認可、暗号化、入力検証に関するコードの変更やレビューが要求されたとき。

## 目的
OWASPトップ10に基づくセキュリティチェックを実行する。

## 手順
1. 変更対象ファイルの特定
2. セキュリティパターンの検証
3. 脆弱性レポートの生成
```

### 当たり前のことは書かない

Claude がデフォルトでできることは記述しません。**デフォルト動作を超えた振る舞い**にフォーカスします。

```markdown
# 悪い例
- コードを読んでください
- エラーがあれば修正してください

# 良い例
- race condition のパターンを検出したら、mutex ではなく channel ベースの解決策を優先する
- SQL クエリを検出したら、N+1 問題の有無を必ず確認する
```

### 手順を押し付けない

ゴールと制約を与え、具体的なステップはClaude に任せます。

```markdown
# 悪い例
1. まずファイルAを開く
2. 次に関数Bを見つける
3. そしてCに変更する

# 良い例
## ゴール
パフォーマンスボトルネックを特定し、改善案を提示する

## 制約
- 既存のAPIインターフェースを変更しない
- メモリ使用量を増加させない
- ベンチマーク結果を添付する
```

## 段階的開示（Progressive Disclosure）

スキルフォルダ内のサブディレクトリを使って、必要なときに必要な情報だけを開示します。

```
.claude/skills/api-design/
├── SKILL.md              # 基本ルール（常に読み込み）
├── references/
│   ├── rest-conventions.md  # 必要時に参照
│   └── graphql-patterns.md  # 必要時に参照
├── scripts/
│   └── openapi-validate.sh  # 検証スクリプト
└── examples/
    ├── crud-api.md          # CRUD APIの例
    └── pagination.md        # ページネーションの例
```

## 動的インジェクション

SKILL.md 内で `` !`command` `` 構文を使うと、スキル呼び出し時にシェルコマンドの出力を動的に注入できます。

```markdown
# デプロイスキル

## 現在の環境情報
!`kubectl get nodes -o wide`

## 最新のデプロイ状態
!`kubectl rollout status deployment/app`
```

Claude がスキルを呼び出した時点でコマンドが実行され、最新の状態が注入されます。

## コンテキストフォーク

`context: fork` を設定すると、スキルがサブエージェントのコンテキストで実行されます。

```yaml
context: fork
agent: security-reviewer
```

- メインコンテキストは**最終結果のみ**を受け取る
- 中間的なツール呼び出しはメインコンテキストに表示されない
- 重い処理をオフロードしてメインコンテキストを軽く保てる

## Gotchas セクション

スキルには**既知の落とし穴と失敗パターン**をまとめた Gotchas セクションを設けます。使い込むほど高シグナルな知見が蓄積されます。

```markdown
## Gotchas

- TypeScript の `strict: true` 環境では `any` 型の使用を拒否されるため、
  必ず具体的な型定義を提供すること
- ESM/CJS の混在環境では import 文の変換に失敗しやすい。
  パッケージの module 設定を先に確認する
- テストファイルのモック設定はプロジェクトごとに異なる。
  既存テストのパターンを必ず参照する
```

## スキル利用状況の計測

PreToolUse フックを使って、どのスキルがどれくらい使われているかを追跡できます。利用率が低いスキルはトリガー記述の改善が必要かもしれません。

## ベストプラクティスまとめ

| 項目 | 推奨 |
|------|------|
| 構造 | ファイルではなくフォルダで構成 |
| 記述 | モデル向けトリガー記述 |
| 内容 | デフォルト動作を超えた振る舞いのみ |
| 指示 | ゴールと制約を与え、手順は任せる |
| 素材 | スクリプトやライブラリを同梱して合成を促進 |
| 失敗知見 | Gotchas セクションを継続的に更新 |
| 計測 | PreToolUse フックで利用状況を追跡 |
