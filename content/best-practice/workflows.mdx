# 開発ワークフロー比較（8パターン）

[Workflows - Claude Code Best Practice](https://github.com/shanraisshan/claude-code-best-practice)

## 概要

Claude Code のエコシステムでは、コマンド・エージェント・スキルを組み合わせた多様なワークフローが登場しています。ここでは代表的な**8つのワークフロー**を比較し、それぞれの特徴と適用場面を解説します。

すべてのワークフローは共通のパターンに収束します：

**Research → Plan → Execute → Review → Ship**

## ワークフロー選択ガイド

### シンプルなタスク

素の Claude Code がもっとも効果的です。複雑なワークフローはオーバーヘッドになります。

### 中規模プロジェクト

Spec Kit や gstack のようなシンプルな構成が適しています。コマンドとスキルの組み合わせで十分です。

### 大規模チーム開発

Everything Claude Code や BMAD-METHOD のような包括的な構成が力を発揮します。

## 共通のワークフロー原則

### プランモードを活用する

実装前に必ず計画を立てます。フェーズごとにゲートを設け、各フェーズで複数種類のテスト（ユニット、自動化、統合）を実行します。

```
> /plan 認証機能の実装

Phase 1: データモデル設計
  ├── ユニットテスト
  └── ゲート: スキーマレビュー

Phase 2: API実装
  ├── 統合テスト
  └── ゲート: セキュリティレビュー

Phase 3: フロントエンド
  ├── E2Eテスト
  └── ゲート: UXレビュー
```

### コンテキスト管理

エージェントの「ダムゾーン」（性能低下領域）を避けるため、コンテキスト使用量が**50%に達したら手動で `/compact` を実行**します。

### モデルの使い分け

- **Opus** — 計画、設計、複雑な推論に使用
- **Sonnet** — コーディング、定型的な実装に使用

`ultrathink` キーワードで高負荷推論を有効にできます。

### セッション管理

- 重要なセッションは名前を付けて保存し、後から再開可能にする
- 複数の Claude インスタンスを並行実行する場合はセッション名で区別する
- `/rewind`（Esc Esc）で取り消し。同じコンテキスト内での修正より効率的
