# トークン最適化・コスト管理

[Everything Claude Code - Token Optimization](https://github.com/affaan-m/everything-claude-code)

Claude Codeの利用コストを最適化するための戦略を解説します。モデル選択、コンテキスト管理、戦略的コンパクションなどの手法を組み合わせることで、品質を維持しながら大幅なコスト削減が可能です。

## 推奨設定

```json
{
  "model": "sonnet",
  "env": {
    "MAX_THINKING_TOKENS": "10000",
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50",
    "CLAUDE_CODE_SUBAGENT_MODEL": "haiku"
  }
}
```

## モデル選択戦略

タスクの複雑さに応じてモデルを使い分けることが、最も効果的なコスト削減手法です。

| タスク | 推奨モデル | コスト削減率 |
|---|---|---|
| 実装・テスト・デバッグ | Sonnet | 約60% |
| 複雑なアーキテクチャ・深い推論 | Opus | フル性能 |
| サブエージェントのレビュー | Haiku | 約90% |

### 日常のワークフロー

```bash
/model sonnet      # Sonnetで開始（タスクの80%をカバー）
/cost              # トークン使用量を監視
/clear             # 無関係なタスク間で無料リセット
/compact           # マイルストーンで戦略的にコンパクション
/model opus        # 必要な時だけOpusに切り替え
```

### モデルルーティング

```bash
/model-route       # タスクの複雑さに応じて自動でモデルを選択
```

## 戦略的コンパクション

コンパクション（`/compact`）のタイミングが品質とコストに大きく影響します。

### コンパクションすべきタイミング

- **リサーチ・探索の後、実装の前** — 探索で得た知見を要約して実装に集中
- **マイルストーン達成後、次フェーズの前** — 完了した作業の詳細を圧縮
- **デバッグセッションの後、続行前** — デバッグの試行錯誤を整理
- **失敗したアプローチの後、新方針の前** — 失敗の詳細を捨て、教訓だけ保持

### コンパクションすべきでないタイミング

- **実装の途中** — 変数名、ファイルパス、コンテキストが失われる
- **アクティブな改良中** — 連続的な変更の文脈が必要
- **デバッグにコンテキストが必要な時** — エラーの履歴や試行錯誤が参考になる

### 自動コンパクション設定

```json
{
  "env": {
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50"
  }
}
```

この設定により、コンテキストの50%使用時に自動コンパクションが発動します。デフォルトの95%より早いタイミングで実行され、よりスムーズな体験が得られます。

ただし、ECCでは自動コンパクションよりも `strategic-compact` スキルによる論理的なタスク区切りでのコンパクションを推奨しています。

## コンテキストウィンドウの管理

### MCPの影響

有効なMCPサーバーのツール説明がコンテキストを消費します：

- MCPなし: 約200kトークンが利用可能
- MCPを多数有効化: 約70kトークンまで縮小

### 最適化方法

```json
{
  "disabledMcpServers": ["supabase", "railway", "vercel"]
}
```

**ガイドライン:**
- 有効なMCPは10個以下に抑える
- アクティブなツールは80個以下に維持
- 使わないMCPは `disabledMcpServers` で無効化

### 思考トークンの制限

```json
{
  "env": {
    "MAX_THINKING_TOKENS": "10000"
  }
}
```

思考トークンを制限することで、応答あたりのコストを抑えます。ただし、複雑な推論が必要な場合は制限を緩めてください。

## サブエージェントのコスト最適化

```json
{
  "env": {
    "CLAUDE_CODE_SUBAGENT_MODEL": "haiku"
  }
}
```

サブエージェントにHaikuを使用することで、レビューや簡単な分析タスクのコストを約90%削減できます。

## コスト考慮パイプライン

ECCの `cost-aware-llm-pipeline` スキルは、タスクの複雑さを自動判定してモデルを選択するパイプラインです：

1. タスクを受け取る
2. 複雑さを分析（単純/中程度/複雑）
3. 適切なモデルにルーティング
4. 結果を返す

これにより、人間が毎回モデルを切り替える手間なく、自動的にコストが最適化されます。

## コンテンツハッシュキャッシュ

`content-hash-cache-pattern` スキルにより、同じ内容のリクエストに対するキャッシュが機能します。同じファイルの分析を繰り返す場合などに、トークン消費を大幅に削減できます。

## コスト監視

```bash
/cost                  # 現在のセッションのトークン使用量を確認
```

定期的に `/cost` でトークン消費を確認し、予想外の増加がないか監視しましょう。

## まとめ：コスト最適化チェックリスト

1. デフォルトモデルをSonnetに設定
2. サブエージェントモデルをHaikuに設定
3. 思考トークンを制限（10,000程度）
4. 不要なMCPを無効化
5. 戦略的タイミングでコンパクション
6. `/clear` で無関係タスク間のリセット
7. `/cost` で定期的に使用量を監視
8. 複雑なタスクのみOpusに切り替え
