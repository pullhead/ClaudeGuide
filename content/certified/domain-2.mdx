# D2: ツール設計 & MCP (18%)

## MCP（Model Context Protocol）の3原則

MCPはClaudeと外部サービスを接続するためのオープンプロトコル。

### Tools（ツール）
- モデルが呼び出す関数（API呼び出し、データベースクエリ等）
- JSON Schemaで入出力を定義
- **description が最重要** — Claudeはdescriptionを読んでツールの使い方を判断する

### Resources（リソース）
- 読み取り専用のデータソース（ファイル、DB内容等）
- コンテキストとしてClaudeに提供される

### Prompts（プロンプト）
- 再利用可能なプロンプトテンプレート
- サーバー側で定義し、クライアントが利用

## ツールスキーマ設計のベストプラクティス

### description の重要性
- Claudeはツールの `description` を見て使うかどうかを判断する
- 曖昧な説明 → 不適切なツール選択
- **いつ使うか**、**いつ使わないか**を明記する

### 最適なツール数
- **4〜5個が最適** — 多すぎるとClaudeが混乱する
- 関連するツールはグループ化して1つのMCPサーバーにまとめる
- ツールが多い場合は段階的に提供する

### パラメータ設計
- 必須パラメータは最小限に
- デフォルト値を活用
- enum型で選択肢を制限する
- 明確な型定義（string, number, boolean等）

## MCPサーバー構成

### ローカルサーバー（stdio）
- ローカルプロセスとして起動
- Claude Desktopやエディタ拡張から利用
- 設定例: `claude_desktop_config.json`

### リモートサーバー（HTTP/SSE）
- リモートAPIとして公開
- 認証・認可の設計が必要

### セキュリティ考慮事項
- ツール呼び出しの権限管理
- 機密情報の取り扱い（APIキー等）
- Human-in-the-loop: 重要な操作には人間の承認を挟む

## 試験のポイント
- MCP の3つの要素（Tools, Resources, Prompts）の違い
- description がツール選択に与える影響
- ツール数の最適化（4-5個が理想）
- セキュリティとHuman-in-the-loopの設計
